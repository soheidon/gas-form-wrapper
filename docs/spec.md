# 院内アンケートシステム 基本仕様書 v1.2

> **v1.1からの改訂。変更箇所には【v1.2変更】を付記。**
> v1.1実装レビュー（ChatGPT）で指摘されたセキュリティ・パフォーマンス・UXの問題を反映。

## 0. 用語・前提

| 用語 | 定義 |
|------|------|
| Form | Googleフォーム（設問の定義・回答受領） |
| Response Sheet | フォームに紐づく回答スプレッドシート |
| WebApp | GAS（HtmlService）で提供するカスタムUI |
| Submit | GASが `FormApp.createResponse().submit()` でフォームへ回答を送信 |
| Op Log | 運用ログ用スプレッドシート |

- 想定環境：Google Workspace（組織内限定運用）
- 想定同時利用者数・設問数の上限を明記（→ §10）

---

## 1. 目的

1. Workspace内で完結し、外部公開を避ける。
2. 回答者にとって入力しやすいカスタムUIを提供する。
3. 回答の保存は「失敗しうる」ことを前提に、失敗検知・再送・監査を備える。
4. 非エンジニアがGoogleフォームで設問を更新できるようにしつつ、更新時の破綻を防ぐ。

## 2. 全体アーキテクチャ

### 2.1 コンポーネント

1. **Googleフォーム**（設問管理・回答受領の正）
2. **GAS Webアプリ**（UI配信・回答受付・Submit実行）
3. **回答スプレッドシート（Response Sheet）**（蓄積・集計・二次利用）
4. **運用ログ用スプレッドシート（Op Log）**（監査・再送キュー・障害解析）

【v1.1追加】**Response SheetとOp Logは必ず物理的に別ファイル（別スプレッドシートID）とする。** Googleフォームが回答を自動転記するResponse Sheetと、GASが頻繁に読み書きするOp Logを同一ファイルの別シートにすると、フォーム内部の行追加処理とGASのLockServiceが裏側で競合し、予期しないエラーや行ズレを引き起こすリスクがある。

### 2.2 データの責務分離

- フォーム／Response Sheet：**最終的な正の記録（Single Source of Truth）**
- Op Log：**途中失敗や再送のための補助記録（可観測性）**

### 2.3 GASデプロイとクライアントの整合

GASコードを更新デプロイした際、既に開いているブラウザタブは旧バージョンのHTMLを保持している。フォーム定義のrevisionだけでなく、WebApp自体のバージョン不整合も起こり得る。

対策：
- WebAppのHTML内にデプロイバージョン（例：ビルド日時）を埋め込む
- 送信時にこのバージョンも送信し、サーバ側で完全一致を判定（許容範囲は持たない）
- 不整合時は「画面を再読み込みしてください」を返す（フォームrevision不一致と同じUX）
- 【v1.2追加】再読み込み時に入力内容が消えることを防ぐため、クライアント側でlocalStorageに下書きを保存する（→ §8.1）

【v1.2追加】**デプロイ時の運用ルール：**
- デプロイ前に利用者へ告知する（「入力中の方は先に送信を完了してください」）
- デプロイ後、スクリプトプロパティの `WEBAPP_VERSION` をインクリメントする
- 開いたままの画面はバージョン不一致エラーになるが、下書き保存により再読み込み後に入力内容が自動復元される

### 2.4 【v1.2追加】クライアント-サーバ通信の方式

Phase 1 ではクライアント-サーバ通信に `google.script.run` を採用する。`doPost` は外部連携・API経由アクセス用の予備として残すが、回答画面・管理画面からの通信は `google.script.run` を正とする。

採用理由：
- GASの認証が自動で効き、CSRF対策やセッション管理の追加実装が不要
- `doPost` + `fetch` ではCORSやリダイレクト周りでGAS特有のハマりどころがある
- `google.script.run` はコールバック形式（withSuccessHandler/withFailureHandler）で、エラーハンドリングが素直に書ける

## 3. データ処理フロー

### 3.1 設問の配信（GET）

1. 回答者がWebAppへアクセス
2. GASがフォーム定義を読み出し、UI生成用JSONを返す
   - `itemId`（`Item.getId()`）
   - `title` / `helpText`
   - `type`
   - `choices`（選択肢）
   - `required`
   - `validation`（可能な範囲 → §3.3で検証範囲を定義）
3. フォーム定義のバージョン情報を付与
   - `formRevisionHash`
   - `publishedAt`（GAS側での生成時刻）
4. `webAppVersion`（デプロイバージョン）も付与

### 3.2 回答送信（POST）

1. ブラウザが回答JSONをGASへ送信
   - 回答値
   - `formRevisionHash`
   - `submissionId`（UUID）
   - `webAppVersion`
2. GASはまず**サーバ側バリデーション**を実行（→ §3.3）
3. バリデーション通過後、Op Logに `RECEIVED` を記録（ロック制御 → §9.2）
4. `FormApp` でSubmitを試行
5. 成功 → Op Logを `SUBMITTED` に更新（ロック制御 → §9.2）
6. 失敗 → Op Logを `FAILED_TRANSIENT` or `FAILED_PERMANENT` に更新（エラー分類 → §6.4、ロック制御 → §9.2）

【v1.1変更】v1.0ではステップ5に「editResponseUrlを記録」と書いていたが、Phase 1では保存しない。`FormResponse.getEditResponseUrl()` が返す編集URLはリンクを知る人が回答を編集できる性質があり、院内システムでは取り扱いリスクが高い。回答編集機能が必要になった場合はPhase 2以降で運用ポリシー（保存先のアクセス制限・利用条件・失効管理）を設計したうえで導入する。

### 3.3 サーバ側バリデーション

**クライアントは信用できない**ため、サーバ側で検証する。

【v1.1変更】v1.0では「フォーム側定義に準拠」と書いていたが、FormAppから取得できるバリデーション情報は設問タイプにより限定的であり、フォーム側の検証ルールを完全に再現することは困難な場合がある。Phase 1ではGAS側で確実に実装できる範囲に限定し、Phase 2でFormItemのValidationオブジェクトを活用した高精度な検証に拡張する。

**Phase 1の検証項目：**
1. `formRevisionHash` の一致確認
2. `webAppVersion` の許容範囲確認
3. `submissionId` の形式チェック（UUID v4形式）
4. 必須項目（`required: true`）の未入力チェック
5. 選択肢型（ラジオ・チェックボックス・プルダウン）の回答値がフォーム定義上の有効な選択肢に含まれるか
6. テキスト型の文字数上限チェック（GAS側で定義する固定上限値。Phase 1では全テキスト型共通で10,000文字）

**Phase 2で追加：**
- 各FormItemの `getValidation()` から取得した正規表現・数値範囲等の適用
- 日付・時刻型の書式検証

バリデーション失敗時は**Op Logに記録せず**、即座にエラーレスポンスを返す（不正リクエストをログに溜めない）。

## 5. フォーム変更への追随

### 5.1 revision hashの生成方法

【v1.1変更】hash長を16文字から32文字に変更。16文字（64bit）でも衝突確率は極めて低いが、短縮する実益が薄く、万一の衝突時にデータ破壊につながるリスクを考慮し、安全側に倒す。

また、選択肢文字列の正規化方針を追加。フォーム編集者が意図しない空白の付加でhashが変わり、回答者が不要な再読み込みを強いられることを防ぐ。

```
// 1. 選択肢文字列の正規化
//    - 前後の空白を除去（trim）
//    - 全角半角の正規化はしない（意図的な使い分けを壊すリスクがあるため）

// 2. hash入力の生成
hash入力 = JSON.stringify({
  items: [
    {
      id: <itemId>,
      type: <itemType>,
      required: <boolean>,
      choices: [<trim済みchoice文字列の配列>]  // 選択肢型のみ
    },
    ...  // itemIdの昇順でソート
  ]
})

// 3. hash生成
formRevisionHash = SHA256(hash入力).substring(0, 32)  // 32文字（128bit）
```

**意図的にhashに含めないもの：**
- `title`、`helpText`（ラベル修正は互換変更とみなす）
- `description`（フォームの説明文）

これにより「安全な変更」ではhashが変わらず、回答者の再読み込みが不要になる。

### 5.2 変更検知と拒否

- 送信時に `formRevisionHash` を照合し、不一致なら **Reject**
- メッセージ：「設問が更新されました。画面を再読み込みしてください。」
- Reject時はOp Logに記録しない（正常なガード動作のため）

### 5.3 互換ポリシー

- **安全（hashが変わらない）**：title/helpText修正、description変更
- **危険（hashが変わる）**：type変更、required変更、選択肢の追加・削除・順序変更、設問の追加・削除

## 6. 送信の安定化

### 6.1 Write-Ahead Log（WAL）パターン

- Op Logに `RECEIVED` を書いてからSubmitを実行する
- Submit失敗時もOp Logに回答データが残るため、再送が可能

### 6.2 再送メカニズム

- **手動再送**：管理者UIからFAILED/STALE_RECEIVEDレコードを選択して再送
- **自動再送**：時間主導トリガー（例：5分間隔）でFAILED_TRANSIENTを再試行
  - 再試行上限：3回
  - 連続失敗時は管理者へメール通知

【v1.1追加】**RECEIVED孤児の自動検知と回収**

GAS特有の問題として、`FormApp.submit()` の実行中にV8エンジンのクラッシュや実行時間制限（6分）の超過が起きた場合、`try-catch` にすら入らず処理が強制終了し、Op Logのステータスが `RECEIVED` のまま永久に更新されないことがある。

対策：
- 自動再送トリガーの対象条件に `FAILED_TRANSIENT` だけでなく、**一定時間（15分）以上 `RECEIVED` のまま放置されているレコード**も含める
- 該当レコードは `STALE_RECEIVED` に遷移させたうえで再送を試行する
- 再試行上限（3回）は `FAILED_TRANSIENT` と共通

### 6.3 再送時のフォーム変更問題

回答受付後〜再送実行までの間にフォーム構造が変わった場合、再送は永久に失敗する。

対策：
- 再送時にもrevision hashを照合する
- 不一致の場合は `FAILED_PERMANENT`（再送不能）に遷移し、管理者に通知
- 管理者は回答データ（Op Logに保存済み）を見て手動でフォームに入力するか、データを別途保全する

### 6.4 エラー分類

一時的エラーと恒久的エラーでは対処が異なる。

【v1.1変更】`STALE_RECEIVED` を追加。

| 状態 | 意味 | 自動再送 |
|------|------|----------|
| `RECEIVED` | 受付済み、Submit処理中 | -（処理中のため対象外） |
| `STALE_RECEIVED` | 受付後15分以上経過、処理が異常終了した可能性 | 対象（上限あり） |
| `SUBMITTED` | 送信成功 | - |
| `FAILED_TRANSIENT` | 一時エラー（タイムアウト、クォータ超過等） | 対象（上限あり） |
| `FAILED_PERMANENT` | 恒久エラー（フォーム削除、revision不一致等） | 対象外 |
| `ABANDONED` | 再試行上限到達 | 対象外 |

### 6.5 冪等性（重複送信対策）

- `submissionId` を必須とし、Op Logで既送信を判定
- 同一 `submissionId` の再送信は二重Submitしない
- `submissionId` はクライアント生成だが、サーバ側でもUUID形式を検証する

## 7. 権限・セキュリティ

### 7.1 Webアプリの実行権限とデプロイ設定

- **「自分（開発者）」として実行**
- Webアプリ閲覧：組織内ユーザーに限定

【v1.1追加】GASデプロイ時の設定の組み合わせを明記する。`Session.getActiveUser().getEmail()` を確実に取得するための前提条件：

| 設定項目 | 値 |
|----------|-----|
| 実行するユーザー | 「自分」（スクリプトオーナー） |
| アクセスできるユーザー | **「Google Workspace 組織内の全員」** |

「全員（匿名を含む）」を選択すると、組織外ユーザーや未ログインユーザーからのアクセスが可能になり、`getActiveUser()` が空文字を返す確率が大幅に上がる。**院内システムでは「組織内全員」を選択すること。**

### 7.1.1 【v1.2追加】管理者認可

管理者画面（?page=admin）および管理者向けサーバ関数（getFailedRecords、getSuccessRate24h、manualRetry等）には、**ページ配信時とサーバ関数実行時の両方で**管理者認可チェックを行う。

- `Session.getActiveUser().getEmail()` を取得し、スクリプトプロパティ `ADMIN_EMAILS` と照合する
- メール未取得（空文字）の場合は拒否する（管理画面は匿名アクセスを許容しない）
- ドメイン内の非管理者がアクセスした場合は「アクセス権限がありません」を返す

**`google.script.run` はドメイン内の誰でも呼べる**ため、ページ配信時（doGet）のチェックだけでは不十分。サーバ関数側でのガードが必須である。`XFrameOptionsMode` はデフォルト（SAMEORIGIN相当）を使用し、iframe埋め込みによるクリックジャッキングを防ぐ。

### 7.2 `getActiveUser()` が空を返すケースへの対策

「組織内全員」設定でも、サードパーティCookieのブロック等により空文字が返ることがある。これは「例外」ではなく**起こりうる事象**として設計する。

【v1.1変更】v1.0では対策が運用ポリシー判断のみだったが、技術的なフォールバック手段を追加。

**ユーザー識別の優先順位：**
1. `Session.getActiveUser().getEmail()` — 取得できればこれを使用
2. `Session.getTemporaryActiveUserKey()` — メール取得不可時のフォールバック。ユーザー固有の一時キー（同一ユーザーなら30日間は同じ値を返す）で、最低限の追跡が可能
3. いずれも取得不可 — `"UNKNOWN"` を記録

**運用ポリシーとの組み合わせ：**
- **記名式アンケート**：優先順位1が取得できない場合、回答を受け付けない（エラーレスポンス）。ただし、フォーム側で「メールアドレスを収集」設定をONにしておけば、`FormResponse.getRespondentEmail()` でSubmit後に取得する代替経路もある（要検証）
- **匿名アンケート**：優先順位2まで取得できればOp Logに記録。3の場合でも回答は受け付ける

### 7.3 フォームの権限

- 回答者にはフォーム編集権限不要
- フォーム編集者（非エンジニア）は、指定のGoogleグループに限定

### 7.4 個人情報・監査

- 収集する個人情報（氏名・ID等）がある場合は以下を明記：
  - 目的
  - 保管期間
  - 参照権限
- Op Logには必要最小限のみ保存する（回答データの保存方法は§9.1参照）

## 8. 画面仕様

### 8.1 回答画面

- 送信ボタン押下後は**二重送信防止**（ボタン無効化＋スピナー表示）
- 【v1.1追加】送信失敗時、クライアント側で `submissionId` と回答データを保持し、「再試行」ボタンを表示する。これにより、ロック取得失敗やネットワークエラー時にユーザーが入力をやり直す必要がなくなる
- 【v1.2追加】**localStorage下書き保存：** 入力のたびにlocalStorageへ下書きを保存する。webAppVersion不一致やブラウザリロード時に入力内容が消えることを防ぐ。
  - キー：`survey_draft`（固定。submissionIdはリロード後に再生成されるため使わない）
  - TTL：2時間（古い下書きを自動的に無効化）
  - 送信成功時・受付済み（Op Log記録あり）時に削除
  - バリデーションエラー・通信エラー時は保持（再入力・再試行に備える）
  - リロード後、下書きが存在すれば自動復元し「前回の入力内容を復元しました」と通知する
- 送信結果の表示：
  - 成功 → 完了画面（submissionId表示）
  - RECEIVED → 「回答を受け付けました。システムが処理中です。」
  - 失敗 → 受付済み（Op Log記録あり）と受付失敗（記録なし）を**明確に区別**
  - 【v1.1追加】ロック取得失敗 → 「混雑しています。しばらく待ってから再試行してください。」（再試行ボタン表示）

### 8.2 管理者画面（Phase 1最小構成）

【v1.2追加】管理者認可必須（§7.1.1参照）。非管理者にはアクセス拒否画面を表示する。

- FAILED一覧表示（STALE_RECEIVED / FAILED_TRANSIENT / FAILED_PERMANENT / ABANDONED）
- 選択した回答の手動再送ボタン（管理者手動の場合、FAILED_PERMANENTやABANDONEDも再試行可能。ただしrevision hash不一致時は不可）
- 直近24時間の成功率サマリ
- 操作結果の通知表示（成功/失敗を5秒間表示）
- 【Phase 2で追加】エラー種別集計、時系列グラフ

## 9. 運用・監視

### 9.1 Op Logの列設計

【v1.1変更】`responseData` の保存方法を変更。Google Sheetsには1セルあたり約50,000文字の上限があり、段落型の設問が複数ある場合や設問数が多い場合に回答JSONが上限を超えるリスクがある。

**Phase 1の方針：** Op Logの `responseData` 列には回答JSONをそのまま保存する。ただし、書き込み前にJSON文字列長を検査し、**40,000文字を超える場合はGoogle DriveにJSONファイルとして保存し、Op LogにはDriveファイルIDのみ記録する**（安全マージンとして10,000文字の余裕を確保）。

【v1.2追加】Drive退避先はスクリプトプロパティ `OPLOG_DRIVE_FOLDER_ID` で指定した専用フォルダとする。回答データを含むファイルがマイドライブ直下に散乱することを防ぎ、アクセス権限を管理者のみに制限する。

| 列 | 内容 |
|----|------|
| `submissionId` | UUID |
| `status` | RECEIVED / STALE_RECEIVED / SUBMITTED / FAILED_TRANSIENT / FAILED_PERMANENT / ABANDONED |
| `receivedAt` | 受付時刻（ISO 8601） |
| `submittedAt` | Submit成功時刻 |
| `formRevisionHash` | 受付時のhash |
| `userEmail` | getActiveUser() の結果（空の場合は次の列を参照） |
| `userTempKey` | 【v1.1追加】getTemporaryActiveUserKey() の結果（フォールバック用） |
| `responseData` | 回答JSON、または `drive:<fileId>` 形式のDrive参照 |
| `errorType` | エラー種別（TIMEOUT / QUOTA / FORM_DELETED / REVISION_MISMATCH / SCRIPT_CRASH 等） |
| `errorMessage` | エラー詳細 |
| `retryCount` | 再試行回数 |
| `lastRetryAt` | 最終再試行時刻 |

### 9.2 Op Logへの同時書き込み制御

GASでスプレッドシートに同時書き込みすると、LockServiceを使わない場合にデータ競合が起こる。

【v1.1変更】LockServiceの適用範囲（スコープ）を明確化。v1.0では「LockServiceで排他制御する」とだけ書いていたが、**ロックの保持期間を間違えるとSubmit実行中（1〜3秒）にロックを握り続けることになり、後続ユーザーが10秒のロック待ちタイムアウトに到達する。** これはGAS実装で最も起きやすいアンチパターンである。

**正しいロックスコープ：スプレッドシート書き込みの瞬間だけロックを取得・解放し、`FormApp.submit()` 実行中は絶対にロックを保持しない。**

```
// ===== 正しい実装パターン =====

// Step 1: RECEIVED記録（ロック区間①）
Lock取得 → Op Logに RECEIVED を追記 → Lock解放

// Step 2: Submit実行（ロックなし）
FormApp.submit() を実行（1〜3秒かかる場合あり）

// Step 3: 結果記録（ロック区間②）
Lock取得 → 該当行を SUBMITTED or FAILED に更新 → Lock解放
```

```
// ===== アンチパターン（絶対にやらない） =====

Lock取得 → RECEIVED → submit() → SUBMITTED更新 → Lock解放
// ↑ submit()の間ずっとロック保持 → 他ユーザーがタイムアウト
```

**ロック取得の詳細：**
- `LockService.getScriptLock()` を使用
- ロック取得タイムアウト：10秒
- ロック取得失敗時：最大2回リトライ（1秒間隔）→ それでも失敗ならクライアントにエラーレスポンスを返す（回答データはクライアント側に保持されている → §8.1の再試行ボタンで再送可能）

**規模の限界：** 50人以上が同時送信するケースではスプレッドシートへの書き込みがボトルネックになる可能性がある。Phase 2以降でFirestore等への移行を検討する。

### 9.3 障害時チェックリスト

1. FAILED / STALE_RECEIVED が増えていないか
2. フォーム側変更があったか（revision mismatch）
3. GAS実行エラー（スクリプトログ確認）
4. 権限エラー（フォームのアクセス権変更）
5. クォータ超過（GASの日次クォータ確認 → §10.1）

## 10. 制約・リスク

### 10.1 GASの制約

【v1.1変更】クォータ値をGoogle公式ドキュメント（developers.google.com/apps-script/guides/services/quotas）に準拠して修正。v1.0ではConsumer（無料Gmail）の値を一部誤記していた。以下はWorkspace環境の値。

| 制約 | Workspace値 | 影響 |
|------|-------------|------|
| スクリプト実行時間 | 6分/実行 | 大量の再送を1トリガーで処理すると超過する可能性。再送バッチは件数を区切って実行すること |
| 日次トリガー実行時間 | 90分/日 | 自動再送の頻度設計に影響（5分間隔×90分 = 1日18回が上限目安） |
| 同時実行 | 30件/ユーザー、1,000件/スクリプト | 通常運用では問題ないが、全社一斉アンケート等で注意 |
| URLフェッチ | 100,000回/日（Workspace） | v1.0で20,000と記載していたのはConsumer値の誤り。Workspaceでは余裕がある |
| スプレッドシート | 1,000万セル/ファイル | Op Logの肥大化に注意（定期アーカイブが必要） |
| セル文字数 | 約50,000文字/セル | responseDataの保存方法に影響（→ §9.1） |

### 10.2 想定規模

設計判断の根拠として明記する：

- 想定同時利用者数：**〜50名**（これを超える場合はOp Logのストレージ見直しが必要）
- 1フォームの設問数上限：**30問**（UI生成JSONのサイズとGET応答時間に影響）
- Op Logの保持期間：**90日**（超過分はアーカイブシートへ移動）

### 10.3 その他のリスク

- フォーム更新直後の送信競合 → revision照合で拒否
- GASデプロイ更新時の画面不整合 → webAppVersion照合で検知
- Op Logの肥大化 → 定期アーカイブ（時間主導トリガー）
- GASの突然死（V8クラッシュ・実行時間超過）→ STALE_RECEIVED検知で回収（→ §6.2）
